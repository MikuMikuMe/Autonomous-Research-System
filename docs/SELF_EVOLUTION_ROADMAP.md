**Verdict:** The Bias Audit Pipeline is a **pipeline**, not a **self-evolving agentic system**. It has Judge + Revision (Observe → Optimize) but lacks:
- Persistent memory (Remember)
- Dynamic code generation for verification
- Config-driven logic (prompts, rules, templates as resources)
- Optimizer loop that improves prompts/solutions over time

## Target Architecture: Self-Evolving Agentic System

### 1. Act → Observe → Optimize → Remember (Autogenesis loop)

```
Act:     Agents produce outputs (detection, mitigation, auditing)
Observe: Judge + Verification Agent evaluate (rule-based + Gemini semantic + code-based)
Optimize: Revision Agent + Optimizer update prompts/claims/sections when feedback indicates failure
Remember: Persist learnings to memory/ (e.g., "accuracy_delta>0 + fpr_delta<0 → use AUC/Recall framing")
```

### 2. Never Hardcode — Config-Driven Logic

- **Prompts** → `configs/prompts/` (trade_off_summary, mitigation_claims, revision)
- **Rules** → `configs/rules/` (inference logic as declarative rules or Gemini-decided)
- **Templates** → Generated by Gemini when data contradicts defaults; fallback = Gemini, not hardcoded strings

### 3. Verification: Gemini Generates Code, Run It

When Judge or a consistency check flags a claim:

1. **Input:** Claim text + relevant data (JSON, metrics)
2. **Gemini:** "Generate Python code that verifies or refutes this claim given the data. Return only executable code."
3. **Sandbox:** Run code in isolated env (subprocess, timeout, no network)
4. **Output:** `{"verified": bool, "evidence": str, "code": str}`

Example: Claim "Mitigation incurs accuracy loss" + data `accuracy_delta=+0.0069`  
→ Gemini generates: `assert accuracy_delta <= 0, "Claim false: accuracy increased"`  
→ Run → AssertionError → `verified=False`, evidence="accuracy_delta is +0.0069"

### 4. Memory Layer

- `outputs/memory/` — Session summaries, claim→outcome mappings, prompt refinements
- On Judge failure: store (claim, data_snapshot, feedback) for future prompt tuning
- Optimizer (future): Use memory to propose prompt/solution updates

## Implementation Phases

### Phase 1: Verification Agent (Gemini → Code → Run) ✅
- `verification_agent.py` — Given claim + data, Gemini generates verification code, run in sandbox
- Orchestrator: When Judge detects claim/data mismatch, invoke Verification Agent before Revision
- Output: `verification_report.json` (claim, code, result, evidence)

### Phase 2: Config-Driven Prompts ✅
- `configs/prompts/trade_off_summary.txt` — Loaded by mitigation_agent
- `configs/prompts/mitigation_claims.txt` — Gemini infers claims; fallback = rule-based
- `config_loader.load_prompt()` — Format templates with kwargs

### Phase 3: Memory + Optimizer ✅
- `memory_agent.py` — `persist_session()`, `persist_event()` → `outputs/memory/`
- Orchestrator: persists session after pipeline; persists event on Judge failure
- `optimizer_agent.py` — Loads memory, Gemini proposes prompt refinements → `outputs/optimizer_proposals.json`
- Runs in research phase after verification_agent

### Phase 4: Full Autogenesis-Style ✅
- **RSPL:** `configs/resources/registry.json` — agents, tools, prompts, envs with version
- **SEPL:** `sep_layer.py` — propose | commit | rollback | status (auditable lineage)
- **Config composition:** `configs/pipeline.yaml` — core_agents, research_agents, memory, optimizer
- **resource_registry.py** — get_agent(), get_prompt(), list_agents()
